01 / ARCHITECTURE: ISOLATION & CONTEXT LIFECYCLE
agent_isolation_rules.json
**SYSTEM SPECIFICATIONS:** - **Blast Radius Containment**: Agents must be bespoke and strictly isolated per application boundary/IAM project. - **Context Exhaustion Control**: Long incidents exhaust model windows. Systems must run background chronicler summaries. - **SME Instance Rotation**: Dynamically spin up fresh instances of agents seeded only with compressed state summaries.
Core Design Requirements
1. Tenant & IAM Separation

Isolating SRE agents per project boundary limits the IAM access permissions of individual agents, blocking cross-tenant security escalation.

2. Context Lifecycle Management

Periodic 3-minute summarization ticks combine with rotating fresh instances of agents to prevent reasoning degradation during long outages.

02 / RUNTIME: ASYMMETRIC SPEED & POLICY ENGINE
Reactive vs Preventive Controls
1. The Asymmetric Speed Gap
API scale-down takes milliseconds, whereas physical container initialization/pulling takes 30-45 seconds.
2. The Limitation of Script Sweepers
Because startup latency is physical, no reactive recovery script loop can restore system availability in time.
3. Declarative Prevention Controls
Enforce baseline states (HPAs & PDBs) at the K8s API server level to block scaling mutations before they execute.
Incident Management Strategy
Reactive Loops (Too Slow)

Waiting for pods to scale to zero, then executing shell scripts to scale them back up. Blocked by container boot latencies.

Preventive Policy (Zero Latency)

Applying Pod Disruption Budgets (PDB) and Horizontal Pod Autoscalers (HPA). K8s API blocks scale-down natively.
